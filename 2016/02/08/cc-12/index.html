上一讲中我们接触负载均衡器，并且通过一个模拟实验进行了基本的体验，这节课我们通过 Azure 来探索一下负载均衡器的工作原理。

学习目标

  1. Identify the basic internal components of a load balancer.
  2. Recognize the various factors that can affect a load distribution strategy.
  3. Recognize how a load distribution strategy can affect the Quality of Service of an application.
  4. Demonstrate the ability to implement a load balancer using the Round-Robin strategy.
  5. Demonstrate the ability to observe and analyze the affect of an incoming load on the resource utilization of a pool of instances, and be able to implement an efficient load distribution strategy to balance resource utilization.
  6. Demonstrate the ability to monitor the health of instances and handle instance failures without dropping requests.

项目简介

Load Balancers - Review

As discussed in earlier project modules, load balancers are an important element in web services. In general, recall that the need for load balancing in computing stems from two basic requirements,

  1. Improved Quality of Service (QoS), which includes several considerations such as throughput, latency requirements among others.
  2. High Availability (HA), which is the property of a service that is available for near 100% of the time when any client tries to access the service.

Over the last two project modules we have used Amazon’s Elastic Load Balancer (ELB) as a means to improve QoS by distributing the load across multiple data center instances (allowing us the ability to scale horizontally) in order to maximize performance while minimizing cost and to handle failures.

In Project 2.1 for horizontal scaling, initially load balancing was done by the load generator. When you added each individual data center instance, the load generator maintained a list of these and tried to route the load to all of the attached data center instances manually. In the last task of Project 2.1, instead of providing all the data center instances to the load generator, we used an ELB which implements its own strategy to distribute the load. This allowed us to scale the number of data center instances we were using at the back end and hide that information from the client (for our purpose, the load generator can be considered as a large pool of clients). This allows us to handle varying amounts of load without needing to update the client application.

In Project 2.2, in addition to distributing load, the load balancer serves an additional purpose of masking failure. By serving as an extra layer on top of the data center instance layer, the load balancer can remove an offending data center instance from service and continue to serve requests, as long as at least one of the data center instances is alive. ELB does this by monitoring a health check endpoint on every data center instance in order to ensure that the data center instances respond to requests.

The ELB seems like a good solution to the above issues, however, it is not straight forward to design and deploy an effective ELB. Choosing the load distribution strategy among the available instances could impact the effectiveness of the load balancing. Let’s discuss this next.

Load Distribution Strategy

As you already know by now, Amazon’s ELB uses a Round-Robin strategy to distribute the load it receives onto the different data center instances attached to it. It does so without regard to the nature of the request or the current load on the server. (If it has instances across availability zones, the ELB does use weights to assign requests to a data center instance closer to the source of the originating request, but we will ignore this for now, since we are only dealing with a single availability zone).

You would have noticed that in the previous project module, the load generator provided a uniform type of load on all the data center instances. What we mean by this is that every request made to a data center instance required a very similar amount of resources on the data center instance (including CPU, Memory, Disk I/O and Network In/Out) in order to service the request. Therefore when the load varies (i.e. the number of requests per second goes up or down), we can manage the change in capacity easily by scaling out or scaling in.

But consider the scenario where the load is not of uniform type. Here are some of the questions you should ask yourself. What happens in the cases where certain requests use a large amount of resources? Will Round-Robin be an effective solution? Will the utilization of all resources be balanced? How could this affect the Quality of Service if one of the data center instances is over-utilized?

In this project module, we will try to address the concerns presented above by implementing a custom load balancing strategy. We will also explore another core concept of load balancers, which is monitoring the health of the data center instances, and ensuring that only healthy data center instances are considered for load distribution.

Note: We use Azure as the main platform for this project. You can receive full credit for this project by completing only the Azure sections. You may earn a 10% bonus by repeating all 4 tasks on AWS as well.
